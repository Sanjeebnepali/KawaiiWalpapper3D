# Shuffle: built-in pack photos editable + custom timer commits live

**Date:** 2026-05-16
**Type:** fix

## Problem

Two follow-up bugs after `changes/024` opened the edit screen for built-in
packs:

1. **Built-in pack photos couldn't actually be edited.** The picker
   appeared to allow add/remove (the grid was visible, count chip read
   `10/10`), but it behaved as static — tapping any catalog cell raised
   the "Collection full" alert and the existing pack photos couldn't be
   removed. Custom collections worked normally.
2. **Custom timer never persisted.** Tapping "Custom" → upgrading dev →
   typing a minute count → tapping "Start" lost the typed value. The
   timer reverted to the previously saved `customMinutes` (or 60 by
   default).

## Root cause

### 1. Picker source missed pack-photo ids

`app/shuffle/[id].tsx` rendered the grid from `searchCatalog` only.
`searchCatalog` is composed from `featured + categories + couple + mood`
photos — it does NOT include the theme-pack photo ids generated by
`getThemePackPhotos(packId, n)` (which produce ids like
`pink-lolita-0`, `chibi-anime-3`, etc.).

When a built-in pack collection was opened, `selectedIds` held 10
`<packId>-N` ids, none of which appeared in the rendered grid. Result:

- The 10 selected pack photos were invisible (no cells to show their
  selected state).
- The count chip read `10/10` → `isFull = true`.
- Every catalog cell rendered with the "+" badge as unselected.
- Tapping any of them hit `Alert.alert('Collection full', …)` because
  `isFull` short-circuited the add path.

The user perceived the album as "static."

### 2. Custom minutes only committed on blur

The `<TextInput>` for custom minutes only fired `onCommitCustomMinutes`
on `onBlur`. If the user typed `90` and immediately tapped "Start
shuffle", Android could process the tap before the input blurred — the
typed value was discarded and `collection.customMinutes` stayed at its
previous value. The timer label also still read just "Custom" with no
indication of the saved value.

## Solution

### 1. Merge selected-but-not-in-catalog photos into the picker

New `pickerSource` memo in `app/shuffle/[id].tsx`:

```ts
const pickerSource = useMemo(() => {
  const catalogIds = new Set(searchCatalog.map((p) => p.id));
  const extras: { id: string; image: string }[] = [];
  for (const id of selectedIds) {
    if (catalogIds.has(id)) continue;
    const p = getPhotoById(id);
    if (p) extras.push({ id, image: p.image });
  }
  return [...extras, ...searchCatalog.map((p) => ({ id: p.id, image: p.image }))];
}, [selectedIds]);
```

The grid renders `pickerSource` (instead of raw `searchCatalog`):

- Selected pack photos appear at the TOP of the grid with their
  numbered badge — immediately visible for deselection.
- Once the user deselects one, the cap drops to `9/10` and any catalog
  cell becomes addable.
- `getPhotoById` already handles arbitrary `slug-N` ids via its regex
  fallback, so no mockData changes needed.

Why prepend (not append)? The user opened the edit screen *to manage
the existing selection* — putting it at the top is faster to find than
scrolling past the whole search catalog.

### 2. Commit custom minutes on every valid change

`app/shuffle/[id].tsx`:

- New `onChangeCustomMinutes(text)` handler that commits to the store
  on every keystroke when the value parses inside `[5, 24*60]`. Loose
  enough that intermediate typing (e.g. "9" → "90") still validates as
  it goes.
- `onCommitCustomMinutes` (the blur handler) is kept as a backstop —
  snaps the input back to the last saved value if the user blurred
  with junk.
- `<TextInput>` `onChangeText` swapped from `setCustomMinutes` to the
  new committing handler.

Visible feedback so the user sees the value stuck:

- Custom timer option label now reads
  `Custom (90 min)` when `collection.customMinutes` is set — replaces
  the bare "Custom" text. Other options still show their static label
  via `t.label`.

## Files changed

- `app/shuffle/[id].tsx`
  - import `getPhotoById` alongside `searchCatalog`
  - add `pickerSource` memo (selected non-catalog photos prepended)
  - grid renders `pickerSource` instead of `searchCatalog`
  - add `onChangeCustomMinutes`, keep `onCommitCustomMinutes` as blur
    backstop
  - `<TextInput>` `onChangeText` → `onChangeCustomMinutes`
  - timer option label shows `Custom (N min)` when `customMinutes`
    is set

## Verification

Pure JS — no native rebuild.

```
npx expo start --clear
```

Then on device:

1. **Theme Packs → tap "Shuffle" on Pink Lolita** — applies a wallpaper,
   navigates to Active.
2. **Back → tap the gear icon on Pink Lolita** — edit screen opens.
   - Photo picker now shows the 10 `pink-lolita-N` photos at the TOP
     of the grid, each with its numbered selection badge (1–10).
   - Tap any of them → it deselects, count drops to `9/10`.
   - Scroll down → tap any catalog item → it adds, count returns to
     `10/10` with that new photo numbered `10`.
3. **Same screen → Shuffle every → tap "Custom"** — premium gate Alert,
   tap "Upgrade (dev)".
   - Custom input row appears.
   - Type `90` → the label updates **immediately** to
     `Custom (90 min)`. Tap somewhere else → still reads `90`.
   - Change to `45` → label updates live to `Custom (45 min)`.
   - Tap **Start shuffle** without blurring — the saved value sticks
     (verify by re-opening the edit screen).
4. **Custom collection** — repeat steps 2-3 for a user-built collection;
   nothing should regress.

## Notes

- **Why not seed the catalog with theme-pack photos globally?** That
  would dump 6 packs × 18 photos into the in-app search results, which
  is unrelated UX surface. The merge happens locally in the edit
  screen — surgical fix.
- **Why prepend, not append?** Pack photos are what the user came to
  edit; surfacing them first reduces scroll friction. Catalog browsing
  is the secondary action.
- **Re-add a deselected pack photo?** After removing one, the user can
  scroll the catalog to pick a replacement; or, to bring back the
  same pack photo, they need to long-press its pack card on Theme
  Packs → that re-runs `ensureBuiltinPackCollection` which … actually
  no — `ensureBuiltinPackCollection` is no-op when the collection
  exists. So removed pack photos are not re-fetchable from the UI today.
  Acceptable — the user already removed it on purpose. Phase 2 follow-up
  if it becomes a complaint: add a "Re-seed from pack" button on the
  edit screen for built-in pack collections.
- **`onChange` vs debounced commit:** the store write itself is
  debounced 250 ms by `schedulePersist`, so committing on every
  keystroke is cheap — at most one fs write per typing burst.
